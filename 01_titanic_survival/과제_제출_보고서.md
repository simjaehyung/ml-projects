### 전처리 방법

#### 기본 정제

| 처리 내용 | 방법 | 이유 |
|---------|------|------|
| Age 누락값 | Title 그룹별 중앙값으로 채움 | 전체 평균보다 "같은 사회적 위치"의 대표값이 더 합리적 |
| Fare 누락값 | 전체 중앙값 | 1개뿐 |
| Embarked 누락값 | 최빈값(S) | 2개뿐 |
| Cabin 컬럼 | 갑판 추출 후 Pclass로 추정 | 77% 누락이므로 직접 사용 불가 |

<!--
- **중앙값(Median)**: 데이터를 크기순으로 나열했을 때 정가운데 값. 이상치에 덜 민감합니다.
- **최빈값(Mode)**: 가장 자주 나타나는 값.
- **Pclass**: 승객 등급(1=1등석, 2=2등석, 3=3등석).
- **Embarked**: 승선 항구(C=Cherbourg, Q=Queenstown, S=Southampton).
-->

#### 범주형 변수 인코딩 (One-Hot Encoding)

<!--
- **범주형 변수(Categorical Variable)**: 성별, 출발지처럼 '종류'를 나타내는 변수. 숫자는 아니지만 의미가 있습니다.
- **One-Hot Encoding**: 각 범주를 0/1 컬럼으로 분리하는 방식. 예: sex_male=1이면 남성, 0이면 여성.
-->

```
sex       → sex_male (0/1)
embarked  → embarked_Q, embarked_S
Title     → Title_Miss, Title_Mr, Title_Mrs, Title_Officer, Title_Royalty
```

머신러닝 모델은 숫자만 입력받기 때문에 문자형 변수를 이진수로 변환했다.

#### 수치 정규화 (StandardScaler)

<!--
- **StandardScaler**: 각 변수를 평균(μ) 0, 표준편차(σ) 1이 되도록 변환하는 정규화 방법.
- **정규화(Normalization)**: 변수들의 스케일을 맞추어 모델이 특정 변수에 과도하게 의존하지 않도록 하는 작업.
-->

```
정규화 공식: x̃ = (x - μ) / σ
```

적용 컬럼: age, fare, FarePerPerson, FamilySize, FamilyPositionScore, SocialConnectionStrength, DeckLevel

규모가 다른 변수들(나이 0~80, 요금 0~512)을 같은 기준으로 맞추지 않으면 큰 숫자가 모델을 지배한다.

---

### 파생 변수 (인간관계 특성 추가)

<!--
- **파생 변수(Derived Variable)**: 기존 변수들을 조합해서 새로 만든 변수.
- **특성(Feature)**: 모델이 예측에 사용하는 입력 변수입니다.
-->

기존 데이터셋이 개인 속성(나이, 성별, 등급)에만 집중한다는 문제를 발견했다. 타이타닉 침몰은 사회적 관계가 생존에 강하게 영향을 미쳤기 때문에, 다음 변수들을 직접 설계했다.

| 변수 | 수식 | 의미 |
|-----|------|------|
| `Title` | Name에서 정규표현식으로 추출 | 사회적 지위 + 나이 + 성별 압축 |
| `FamilySize` | SibSp + Parch + 1 | 함께 이동해야 하는 인원 수 |
| `IsAlone` | FamilySize = 1이면 1 | 도움받을 사람이 없는지 여부 |
| `FarePerPerson` | Fare ÷ 티켓 공유 인원 | 실제 경제력 (그룹 요금 분담 보정) |
| `FamilyPositionScore` | Title 점수 + 나이 점수 + 가족크기 점수 | 구명보트 탑승 우선순위 수치화 |
| `SocialConnectionStrength` | 0.3×FamilySize + 0.2×TicketGroupSize + 0.5 | 사회적 연결 강도 |
| `DeckLevel` | Cabin 첫 글자 → 1~7 수치화 | 갑판 높이 (A=7 최상, G=1 최하) |
| `MarriedManSacrificeIndex` (MMSI) | 0.5×HasWifeAboard + 0.3×Parch + 0.2×YoungChildrenProxy | 기혼 남성의 "Women & Children First" 희생 패턴 수치화 |

<!--
- **SibSp**: 동승한 형제/자매·배우자 수.
- **Parch**: 동승한 부모·자녀 수.
- **정규표현식(Regular Expression)**: 문자열 패턴을 찾기 위한 특수 문법. 예: "Mr.", "Mrs." 추출.
-->

`FamilyPositionScore`는 역사적 사실인 "Women and Children First" 규칙을 직접 수식으로 표현한 것이다.

## 2. 모델 구현 및 비교

### 사용한 3가지 모델

<!--
- **Logistic Regression**: 선형 회귀의 확률 버전. S자(시그모이드) 함수로 0~1 확률을 출력합니다.
- **Random Forest**: 여러 개의 결정 트리를 학습·투표해 예측하는 앙상블 모델입니다.
- **결정 트리(Decision Tree)**: if-else처럼 질문을 반복해 분류하는 트리 구조 모델입니다.
- **SVM(Support Vector Machine)**: 두 클래스를 가장 잘 나누는 경계선(초평면)을 찾는 모델입니다.
- **베이스라인(Baseline)**: 비교·평가의 기준이 되는 기본 모델입니다.
-->

| 모델 | 작동 원리 | 특징 |
|-----|---------|------|
| **Logistic Regression** | 선형 조합을 S자 함수로 변환해 확률 계산 | 빠르고 해석 용이, 베이스라인 기준 |
| **Random Forest** | 수백 개의 결정 트리를 만들어 다수결로 예측 | 안정적, 특성 중요도 제공 |
| **SVM** | 고차원 공간에서 최적 경계선 탐색 | 소규모 데이터에 강함 |

<!--
- **Train/Test Split**: 데이터를 학습용·평가용으로 나누는 것. 80/20은 80% 학습, 20% 테스트로 나눈 것입니다.

## 3. AutoML (TPOT)

<!--
- **AutoML**: 전처리·모델 선택·하이퍼파라미터를 자동으로 탐색하는 기술입니다.
- **TPOT**: Tree-based Pipeline Optimization Tool. 유전 알고리즘으로 최적 파이프라인을 찾습니다.
- **하이퍼파라미터**: 모델의 성능을 조절하는, 사람이 설정하는 파라미터입니다.
-->

### 설정

```python
TPOTClassifier(
    max_time_mins = 10,    # 탐색 시간 최대 10분
    cv            = 5,     # 5-Fold 교차 검증
    random_state  = 42,    # 재현성 확보
    n_jobs        = 1      # Windows 환경에서 병렬처리 비활성화
)
```

- 탐색 방식: 유전 알고리즘(Genetic Programming) — 파이프라인을 유전자처럼 다루어 세대를 거치며 성능이 높은 조합만 생존시키는 방식
- 탐색 범위: 전처리 방법 + 분류 알고리즘 + 하이퍼파라미터를 자동으로 탐색
- 결과: 3가지 수동 모델 중 최고인 RF(AUC 0.901)에 근접하거나 약간 높은 수준의 파이프라인 자동 발견

<!--
- **파이프라인(Pipeline)**: 전처리 → 모델 학습 → 예측까지를 한 흐름으로 연결한 구조입니다.
- **PCA(Principal Component Analysis)**: 차원 축소 기법. 변수들을 줄여서 계산 효율을 높입니다.
-->

AutoML의 핵심 역할은 사람이 생각하지 못한 조합을 발견하는 것이다. 예를 들어 "데이터 정규화 → PCA 차원 축소 → Extra Trees 분류"처럼 직접 시도하지 않을 파이프라인이 더 좋을 수 있다.

## 4. Random Seed 변경 실험

### 목적

동일한 알고리즘이지만 데이터를 다르게 나눌 때 성능이 어떻게 달라지는지 확인한다. 성능이 크게 흔들리면 그 결과는 "우연"에 의한 것일 가능성이 있다.

<!--
- **Random Seed**: 난수 생성의 시작값. 같은 seed면 같은 난수 순서가 나와 재현성이 생깁니다.
- **분산(Variance)**: 값들이 평균에서 얼마나 흩어져 있는지. 낮을수록 결과가 일관적입니다.
-->

### 결과 (Random Forest, Titanic5 기준)

| Seed | Accuracy |
|------|---------|
| 42 | 0.840 |
| 100 | 0.828 |
| 2026 | 0.836 |
| **평균** | **0.835** |
| **분산** | **0.000034** |

분산이 0.000034로 매우 낮다. 세 번 실행 모두 0.82~0.84 사이로, 결과가 특정 데이터 분할에 의존하지 않는다는 것을 의미한다.

## 5.5 MarriedManSacrificeIndex (MMSI) 실험

"Women & Children First" 규칙에 따른 기혼 남성의 희생 패턴을 수치화한 **MarriedManSacrificeIndex**를 추가하여 성능 변화를 실험했다.

### 과학적 근거
- Frey, Savage & Torgler (2011, *PNAS*): 기혼 남성에서 규칙 준수율이 통계적으로 높음
- 본 데이터: 배우자 동반 탑승 남성(Mr+Mrs) 생존율 **16.9%** (전체 평균 41.6% 대비 −24.7%p)

### MMSI 수식
```
MMSI = 0.5×HasWifeAboard + 0.3×Parch + 0.2×YoungChildrenProxy
(Title≠Mr 또는 독신인 경우 MMSI=0)
```

### 결과 (Δ = +MMSI − Baseline)
| 모델 | ΔAccuracy | ΔAUC-ROC | 평가 |
|------|----------|----------|------|
| Logistic Regression | +0.0038 | +0.0018 | 미세 향상 |
| **Random Forest** | **+0.0115** | **+0.0067** | 유의미한 향상 |
| SVM | +0.0000 | −0.0080 | 소폭 하락 |

Random Forest에서 MMSI 추가 시 AUC-ROC가 0.901 → 0.908로 향상되었다. 기존 피처와 중복되지 않는 새로운 설명력을 추가한 것으로 해석된다. 상세: `reports/SACRIFICE_EXPERIMENT_REPORT.md`

## 6. 핵심 질문 답변

### Q1. 가장 성능이 좋은 모델은?

지표에 따라 달라진다.
- **Accuracy 기준: SVM** (0.847)
- **AUC-ROC 기준: Random Forest** (0.901)

AUC-ROC가 더 본질적인 지표(생존/사망 구별 능력)이므로, **Random Forest를 최고 성능 모델로 선택한다.**

이 데이터는 사망 62% : 생존 38%의 불균형 데이터이기 때문에 단순 Accuracy는 편향될 수 있다. 아무것도 생존하지 않는다고 예측해도 62%의 정확도가 나오기 때문이다.

### Q2. 가장 안정적인 모델은?

**Random Forest** (CV 표준편차 ±0.018, Seed 분산 0.000034)

SVM은 CV 표준편차가 ±0.023으로 가장 높았다. 데이터가 조금 달라지면 성능이 크게 바뀔 수 있다는 의미다. Random Forest는 수백 개의 나무가 서로 오류를 보완하기 때문에 구조적으로 안정적이다.

### Q3. 성능과 안정성이 동일한 모델인가?

이 경우는 **같다 — Random Forest.**

하지만 일반적으로는 다르다. 모델이 복잡할수록 학습 데이터에 과적합되어 성능은 높지만 안정성은 낮아지는 경향(Overfitting)이 있다. Random Forest는 앙상블 구조 덕분에 예외적으로 두 가지를 동시에 갖춘 경우다.

- **앙상블(Ensemble)**: 여러 모델을 조합해 하나의 예측을 내는 방식. 다수결·평균 등.

### Q4. 이 모델을 실제 서비스에 바로 사용할 수 있는가?

**아니다.**

이 모델을 그대로 서비스에 배포할 수 없는 이유:

1. **정적 데이터**: 이 모델은 1912년 데이터를 학습했다. 현실의 재난 상황은 완전히 다른 변수를 가진다.

2. **API가 없다**: 예측을 요청하고 받을 수 있는 서버가 없다. 실무에서는 FastAPI나 Flask로 HTTP 서버를 구축해야 한다.

3. **모델 지속 관리가 없다**: 시간이 지남에 따라 입력 데이터 분포가 변하면 모델 성능이 저하된다(Data Drift). 모니터링과 재학습 체계가 필요하다.

<!--
- **Data Drift**: 시간이 지나며 입력 데이터의 분포가 바뀌는 현상. 모델이 학습한 패턴과 실제 데이터가 달라져 성능이 떨어집니다.
-->

4. **불확실성 미처리**: 모델이 0.51 vs 0.49처럼 거의 확신 없이 예측할 때 어떻게 처리할지 정의되어 있지 않다.

현재 상태에서는 오프라인 분석과 연구 목적으로만 사용 가능하다.

## 7. 내가 정확히 이해하지 못하는 개념 5가지

과제를 수행하며 AI의 설명을 듣고 돌아가게는 했지만, 스스로 완전히 이해했다고 말할 수 없는 개념들이다.

---

**① AUC-ROC의 수학적 의미**

AUC-ROC가 "곡선 아래 넓이"라는 것은 알겠다. ROC 곡선이 True Positive Rate(TPR)과 False Positive Rate(FPR)의 관계를 그린다는 것도 알겠다.

모르는 것: 임계값(threshold)을 바꾸면 왜 TPR과 FPR이 함께 변하는지, 그리고 AUC=0.9가 AUC=0.8보다 구체적으로 얼마나 더 나은 것인지를 직관적으로 설명하지 못한다.

<!--
- **TPR(True Positive Rate)**: 실제 양성 중 맞게 양성으로 예측한 비율 = Recall.
- **FPR(False Positive Rate)**: 실제 음성 중 잘못 양성으로 예측한 비율.
- **임계값(Threshold)**: 확률을 0/1로 바꿀 때의 기준. 0.5보다 크면 생존, 작으면 사망 등.
-->

---

**② Random Forest의 "앙상블"이 내부적으로 어떻게 작동하는가**

수백 개의 나무가 투표한다는 것은 이해한다. 하지만 각 나무가 무작위로 선택되는 데이터(Bootstrap Sampling)와 특성(Feature Subsampling)이 정확히 어떤 기준으로 선택되는지, 그리고 그것이 왜 과적합을 줄이는지를 수식으로 설명하지 못한다.

<!--
- **Bootstrap Sampling**: 원본 데이터에서 중복 허용으로 랜덤 추출해 각 나무마다 다른 학습 데이터를 만드는 방식.
- **Feature Subsampling**: 각 나무가 전체 특성 중 일부만 사용하도록 제한하는 방식. 다양성 확보.
-->

---

**③ StandardScaler 정규화가 실제로 모델에 어떤 영향을 미치는가**

정규화를 안 하면 큰 숫자가 모델을 지배한다고 배웠다. SVM에는 반드시 필요하다고 알고 있다.

모르는 것: Random Forest는 트리 기반 알고리즘이라 정규화가 필요 없다고 어디선가 봤는데, 이 프로젝트에서는 RF에도 정규화를 적용했다. 실제로 RF에서 정규화가 영향을 주는지 안 주는지, 줄 때와 안 줄 때 결과를 직접 비교해보지 않았다.

---

**④ Stratified K-Fold의 "계층화"가 왜 일반 K-Fold보다 나은가**

생존(38%) : 사망(62%)의 비율을 각 폴드에서 유지한다는 설명은 이해했다. 하지만 실제로 Stratified를 쓰지 않으면 얼마나 결과가 달라지는지, 비율 불균형이 어느 정도여야 Stratified를 반드시 써야 하는지 명확한 기준을 모른다.

---

**⑤ 데이터 누출(Data Leakage)의 경계**

이 프로젝트에서 `FamilySurvivalRate`를 전체 데이터에서 계산했더니 정확도가 99%로 올라간 것을 직접 경험했다. 이것이 데이터 누출이라는 것은 이해했다.

모르는 것: 어디까지가 누출이고 어디까지는 괜찮은가. 예를 들어, `FamilySize = SibSp + Parch + 1`도 결국 생존과 관련 있는 가족 정보를 쓰는 것인데 이건 왜 누출이 아닌가. 누출의 정확한 정의와 경계를 스스로 판단하지 못한다.

<!--
- **데이터 누출(Data Leakage)**: 학습 시 '미래 정보'나 '정답에 직접 연결된 정보'를 쓰는 것. 예측 시점에 알 수 없는 정보를 쓰면 누출입니다. FamilySize는 승선 시점에 알 수 있으므로 괜찮고, FamilySurvivalRate는 정답(생존 여부)을 포함하므로 누출입니다.
-->
